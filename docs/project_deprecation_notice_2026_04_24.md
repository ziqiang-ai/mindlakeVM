# MindLakeVM 项目废弃说明

- 状态：Deprecated
- 日期：2026-04-24
- 适用范围：`/Users/mindlake/Documents/mindlakeVM`

## 1. 决策说明

自 2026-04-24 起，`MindLakeVM` 项目进入废弃状态，不再继续投入新的产品化开发工作。

本次废弃不是因为单一技术缺陷或短期资源不足，而是因为项目在产品定义、系统边界、实现可信度三个层面同时存在结构性问题。继续在当前基础上迭代，预计只能增加更多概念、页面和模块，无法有效提高项目的清晰度、可验证性和可交付性。

## 2. 废弃原因

### 2.1 核心功能不清晰

项目长期同时使用了 `MindLakeVM`、`MindLakeOS`、`Doc2Skill`、`RNET`、`Bench` 等多套叙事与命名，导致以下问题：

- 无法用一句稳定的话说明项目的核心功能是什么。
- 对内实现和对外表达不一致，读者难以判断“主功能”与“支撑能力”的边界。
- UI、README、Specs、设计文档都在讲不同层次的内容，但缺少一个最终收束点。

直接结果是：项目看起来内容很多，但真正想解决的问题、主要用户是谁、交付物是什么，并没有被明确固定下来。

### 2.2 关键模块实现缺乏验证性

项目强调“可执行、可验证、可追溯”，但关键运行链路并未形成足够强的工程闭环，主要表现在：

- 编译结果高度依赖 LLM 生成，缺少稳定的结构抽取与一致性保障。
- `trace`、`evidence`、`validation` 等关键输出在很大程度上是后处理结果，而非真实执行过程中的一手产物。
- “验证”更多停留在输出检查和示意层，尚未形成对核心模块行为的强约束。
- 项目宣称的运行时能力与实际主执行路径之间存在落差，导致“可验证性”主要停留在文案和接口字段上。

这意味着项目最重要的价值主张没有被代码层充分证明。

### 2.3 实现边界混乱，代码难以理解

当前代码组织存在明显的职责混杂问题：

- 核心模块同时承担模型定义、提示词组织、文件生成、验证拼装、工具接入等多重职责。
- 模块名和目录结构表面上分层明确，但实际实现中跨层耦合较重。
- 阅读主流程时需要在概念模型、LLM prompt、打包逻辑、运行逻辑和演示逻辑之间频繁切换。

结果是：

- 新读者难以快速建立系统心智模型。
- 维护者难以判断某个改动应属于“产品能力”“运行时逻辑”还是“演示层包装”。
- 后续重构成本不断上升。

### 2.4 文档体系过多，但没有形成真正的单一真源

项目存在 `README`、`docs/`、`design/`、`specs/`、`research/` 等多套文档来源，理论上覆盖全面，实际上造成了认知负担：

- 文档很多，但优先级不清。
- 规范、实现、演示口径并未完全统一。
- “单一真源”更多是目标，而不是已实现状态。

文档数量的增加没有降低理解成本，反而放大了项目定义不稳的问题。

### 2.5 项目更像概念验证堆叠，而不是可持续产品

从当前状态看，项目更接近多个想法的并置：

- 文档编译
- Skill 打包
- Guardrail 执行
- Trace / Evidence 展示
- Bench 对比
- Tool / MCP 扩展

这些方向分别都有讨论空间，但在当前项目中尚未收束为一个简洁、可落地、可反复验证的产品核心。因此继续扩展模块只会进一步增加复杂度，而不会提高系统可信度。

## 3. 结论

本项目废弃的核心原因可以概括为两点：

1. 核心功能没有被清晰定义，系统目标长期漂移。
2. 关键模块没有建立足够强的可验证实现，无法支撑项目的核心承诺。

因此，当前仓库不适合作为正式产品基础继续演进。

## 4. 后续建议

如需在未来重启类似方向，建议不要直接延续当前实现，而应按以下原则重新立项：

### 4.1 先收敛产品定义，再写系统

- 只保留一句核心定义。
- 明确主用户、主输入、主输出、主成功指标。
- 明确哪些能力是核心链路，哪些只是辅助能力。

### 4.2 先做可验证最小闭环

- 优先构建一个小而硬的主流程。
- 关键输出必须能被真实过程直接支撑，而不是事后拼装。
- 在证明核心能力之前，不扩展过多理论层和展示层模块。

### 4.3 重新建立代码边界

- 将“领域模型”“编译逻辑”“运行逻辑”“验证逻辑”“演示层”严格拆开。
- 让主流程能在少量入口文件中被读懂。
- 避免单个模块承担过多角色。

### 4.4 将文档收敛为单一入口

- 只保留一个对外主说明。
- 规格、设计、研究材料必须有明确主从关系。
- 任何文档都应服务于实现清晰度，而不是增加概念密度。

## 5. 归档建议

建议将本仓库视为一次概念探索归档，保留其作为以下用途的参考价值：

- 产品定义失焦案例
- “概念先行、实现滞后”项目的复盘样本
- 未来重新设计相关方向时的反面教材与素材来源

不建议继续在当前代码基础上做增量式修补。
