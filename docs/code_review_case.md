# PR 代码审查案例

## 场景目标

当用户要求审查一个 Pull Request 时，系统应：
- 获取 PR 上下文和差异信息
- 按团队代码审查规范识别高风险问题
- 输出结构化 findings 草稿
- 在 simulate 模式下仅生成草稿
- 在 live 模式下允许真正发布 review 结果

## 审查重点

- 数据库迁移是否具备回滚路径
- API 变更是否保持兼容
- 权限和校验逻辑是否有 negative test
- 测试是否覆盖高风险改动

## 流程

### Step 1：收集 PR 上下文
- 读取仓库路径
- 获取 PR 编号
- 理解用户指定的审查重点

### Step 2：执行代码审查
- 获取 changed files / diff
- 检查数据库迁移风险
- 检查接口兼容性风险
- 检查测试缺口

### Step 3：生成审查草稿
- 输出 findings 列表
- 说明风险等级
- 列出文件位置和理由

### Step 4：发布审查结果
- 仅在 live 模式执行
- 发布前必须显式确认

## 风险控制

- simulate 模式下不得真正发布 review comment
- live 模式必须显式确认
- 若未开放副作用权限，不得执行发布动作

## 期望输出

- 一份结构化代码审查结果
- 每条 finding 包含标题、严重度、文件位置和理由
- 在 live 模式下返回发布结果与审查产物路径
