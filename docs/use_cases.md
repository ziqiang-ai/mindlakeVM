# MindLakeVM 场景应用指南

MindLakeVM 的核心价值是**将企业文档编译为可执行的、有合规约束的 AI Skill**，通过 MCP 协议暴露给任何 AI 工具（Windsurf、Cursor、Claude Code、Claude Desktop）直接调用。

与 RAG 检索或普通 prompt 不同，MindLakeVM 编译出的 Skill 具备：
- **RNET 四核 IR**：结构化的领域知识表示，而非原始文本片段
- **Guardrail 门禁**：硬约束在执行前强制检查，不可绕过
- **执行 Trace**：每个步骤的执行状态和决策记录，支持审计
- **Evidence 引用**：输出结果附带可溯源的文档引用

---

## 场景一：企业 SOP 执行助手

**适合团队**：运维、客服、IT 支持、HR 等拥有标准化操作流程的团队

### 问题背景

企业 SOP 文档通常是 PDF 或 Wiki 页面，AI 工具只能"参考"，无法保证：
- 关键步骤不被跳过
- 高风险操作前必须经过授权检查
- 执行过程可追溯、可审计

### MindLakeVM 方案

```
SOP 文档 (Markdown/PDF)
       ↓ mindlake_compile
  Skill (含 RNET IR + 硬约束)
       ↓ mindlake_run
  执行结果 (含 trace + evidence + validation)
```

**示例：Sev1 事故响应**

将 Sev1 响应 SOP 编译为 Skill 后，运维工程师在 Windsurf 中直接问：

> "线上 k8s Pod 频繁 OOM Killed，影响 3 个核心服务，怎么处理？"

Skill 执行保证：
1. 告警确认步骤不可跳过（硬约束）
2. 影响范围评估前必须通知 oncall（硬约束）
3. 每个步骤输出 trace，留存事后复盘依据
4. 引用 SOP 原文作为操作依据（evidence）

**典型硬约束示例**：
- `禁止在未评估影响范围前执行回滚`
- `P0 告警必须在 5 分钟内响应并通知 oncall`
- `生产环境变更必须有变更单号`

## 场景二：代码规范合规检查

**适合团队**：有安全规范、架构红线、代码审查标准的研发团队

### 问题背景

代码规范文档写得再详细，工程师也难以在编码时时刻对照。传统 linter 只能检查语法，无法检查设计层面的合规性。

### MindLakeVM 方案

将安全规范、架构规范编译为 Skill，在 Cursor/Windsurf 的 Agent 模式中挂载：

**示例：安全编码规范检查**

> "帮我 review 这段代码，检查是否违反了安全规范。"

Skill 的 guardrail 层自动检测：
- 明文硬编码密钥或 token
- 直接拼接 SQL（SQL 注入风险）
- 未做输入校验就调用外部 API

触发硬约束时，AI 直接拒绝并给出具体违规原因，而非给出"建议考虑安全性"这类模糊回复。

**与普通 prompt 的对比**：

| | 普通 prompt | MindLakeVM Skill |
|--|------------|-----------------|
| 约束来源 | AI 临时生成 | 编译自企业规范文档 |
| 可重复性 | 每次回答可能不同 | 相同输入，约束检查结果一致 |
| 审计 | 无 | trace + evidence 完整记录 |
| 维护 | 修改 prompt | 重新编译文档，一处改、处处生效 |

## 场景三：AI IDE 的领域专家工具

**适合场景**：在 Windsurf Cascade 或 Cursor Agent 中需要长期遵守特定领域规范

### 问题背景

给 AI 写很长的 system prompt 来注入领域知识，存在以下问题：
- 上下文窗口有限，长 prompt 影响输出质量
- 规范更新后需要手动同步 prompt
- 多人使用时规范不一致

### MindLakeVM 方案

把 PRD、API 设计规范、数据库命名规范、架构决策记录（ADR）编译为 Skills，挂载到 MCP。

**示例：API 设计规范**

工程师在 Windsurf 中设计新接口时：

> "帮我设计一个用户登录接口"

Cascade 自动调用 `mindlake_run`（API 设计规范 Skill），输出结果自动符合：
- 公司 REST 命名规范（如版本前缀 `/v1/`）
- 认证方式约定（Bearer Token）
- 错误码规范（统一的 `error_code` + `message` 结构）
- 限流策略要求

规范文档更新后，只需重新 `mindlake_compile`，所有 AI 工具立即生效。

## 场景四：多 Agent 协作的共享知识层

**适合场景**：企业内有多个 AI 工具并行使用（Windsurf、Cursor、Claude Code、内部 Bot）

### 问题背景

每个 AI 工具都有自己的上下文，企业知识分散在各处 prompt 中，难以统一维护。

### MindLakeVM 方案

MindLakeVM MCP Server 作为**统一 Skill 仓库**，所有 Agent 通过同一个接口访问：

```
┌─────────────────────────────────────────┐
│         MindLakeVM MCP Server           │
│   (统一编译、统一存储、统一 guardrail)      │
└──────┬───────────┬──────────┬───────────┘
       │           │          │
  Windsurf    Cursor     Claude Code
  Cascade     Agent      (claude mcp)
```

**优势**：
- **一次编译，多处复用**：不需要给每个 AI 工具重复写 prompt
- **版本统一**：规范更新后所有工具同步生效
- **`mindlake_skill_tool_definition`**：可以将 Skill 导出为标准 Claude Tool Definition，注册到任意 Agent

## 场景五：知识密集型文档问答

**适合场景**：RFC、技术方案、政策文件、法规文档的精准问答

### 问题背景

普通 RAG 检索存在的问题：
- 向量相似度检索到的内容可能不是最相关的段落
- AI 可能混合多个不相关段落生成"看起来合理"的错误答案
- 回答缺乏可溯源的引用

### MindLakeVM 方案

MindLakeVM 对文档做的是**语义结构提取**（RNET IR），而非切片后向量化：

- **R 核（表示）**：提取文档的领域、类型、语义空间
- **N 核（标注）**：提取输出结构约束、引用文件
- **E 核（熵控）**：提取硬约束（红线）和软约束
- **T 核（变换）**：提取执行路径和决策树

执行时通过 `cite_reference` tool 显式引用原文，每条输出都有来源支撑。

**适合的文档类型**：
- 金融合规政策（"本次操作是否符合 XX 监管要求？"）
- 医疗 SOP（"XX 情况下的处置流程是什么？"）
- 技术 RFC（"这个设计是否符合 RFC-042 的要求？"）
- 法务合同条款（"合同中关于违约责任的条款是什么？"）

---

## 不适合的场景

| 场景 | 原因 |
|------|------|
| 创意写作、头脑风暴 | 不需要合规约束，普通 AI 即可 |
| 实时数据查询（股价、监控指标） | 需要接外部数据源，MCP 本身不提供实时数据 |
| 非结构化闲聊 | 没有对应 SOP 可编译 |
| 代码补全 | 延迟敏感，MindLakeVM 执行链路较长 |

---

## 快速验证某个场景是否适合

判断标准：
1. **是否有文档可编译？** — 有 SOP、规范、政策文档
2. **是否有"红线"约束？** — 有不可绕过的检查项
3. **是否需要审计记录？** — 需要 trace 留存
4. **是否会被多人反复使用？** — 值得一次编译、多次复用

满足 2 条以上，MindLakeVM 的收益就值得投入。
