\### 1.2 认知协议是什么？ \*\*认知协议\*\*不是普通的提示词，而是一套完整的"思维操作系统"：

    yaml
    认知协议结构:
      元认知层:
        - 角色定义: "我是谁，我的能力边界是什么"
        - 认知框架: "我用什么框架理解问题"
        - 价值标准: "我如何判断好与坏"
        
      方法论层:
        - 问题诊断: "如何识别问题类型"
        - 任务拆解: "如何将复杂任务分解"
        - 决策逻辑: "在什么条件下做什么选择"
        
      执行层:
        - 工作流程: "Step by Step怎么做"
        - 输出规范: "产出物应该是什么形式"
        - 质量标准: "如何自检结果"
        
      边界处理层:
        - 异常识别: "什么情况超出能力范围"
        - 降级策略: "超出范围时怎么办"
        - 求助机制: "何时需要人类介入"

\*\*关键洞察\*\*：普通提示词告诉AI"做什么"，认知协议告诉AI"怎么思考"。 ### 1.3 为什么叫"编译"而不是"提取"或"转化"？ 选择"编译"这个隐喻，是因为它准确捕捉了三个关键特征： \*\*1. 结构化转换\*\* - 不是简单复制粘贴，而是从一种结构转换为另一种结构 - 源知识（人类认知）→ 目标协议（AI可执行） \*\*2. 优化过程\*\* - 编译器会优化代码，认知编译也会优化知识表达 - 消除歧义、显式化隐性规则、增加边界处理 \*\*3. 跨平台可移植\*\* - 编译后的字节码可以在不同机器上运行 - 认知协议可以在不同LLM上执行（GPT、Claude、Llama...） --- ## 第二部分：传统Agent范式的局限 ### 2.1 当前主流Agent架构

\### 3.2 三层架构详解 #### 第一层：Meta-Agent（元Agent） \*\*功能\*\*：将人类知识编译为认知协议 \*\*核心能力\*\*： - 识别文档类型（规范/流程/方法论/案例） - 提取认知结构（步骤/判断条件/质量标准） - 生成标准化协议（统一格式，可组合） \*\*输入输出\*\*：

    输入：任何形式的人类知识文档
    输出：结构化的认知协议模块

\*\*这是"生产Agent的Agent"\*\*——一个Meta-Agent可以批量生成专业Agent。 #### 第二层：Protocol Layer（协议层） \*\*功能\*\*：存储和组合认知协议 \*\*组件\*\*： 1. \*\*协议库\*\*：存储编译好的认知协议模块 2. \*\*协议组合引擎\*\*：根据任务需求组装协议 3. \*\*协议版本管理\*\*：追踪协议的演化和迭代 \*\*协议类型\*\*： | 协议类型 | 功能 | 示例 | |---------|------|------| | 诊断协议 | 识别问题类型 | "这是技术问题还是管理问题？" | | 分析协议 | 深入理解问题 | "五维思维分析框架" | | 决策协议 | 做出判断 | "风险评估决策树" | | 执行协议 | 完成任务 | "代码审查SOP" | | 元协议 | 协调其他协议 | "什么情况下用什么协议" | #### 第三层：Execution Layer（执行层） \*\*功能\*\*：按照协议执行任务 \*\*组件\*\*： - LLM：通用推理能力 - Tools：外部能力接口 - Memory：上下文和历史 - RAG：知识检索 \*\*关键变化\*\*：执行层不再"自由发挥"，而是\*\*严格按照协议层的指令行动\*\*。 ### 3.3 核心机制：协议注入 \*\*传统方式让Agent变强\*\*：

    训练/微调 → 改变模型权重 → 能力固化在模型里

\- 成本高 - 周期长 - 难以快速迭代 \*\*认知编译方式让Agent变强\*\*：

    编译协议 → 注入提示层 → 能力存在于协议里

\- 成本低（写提示词） - 即时生效（无需训练） - 可热更新（随时替换协议） \*\*这意味着\*\*：

    同一个LLM底座：
    
    + 医疗诊断协议 = 医疗Agent
    + 法律分析协议 = 法律Agent  
    + 财务审计协议 = 财务Agent
    + 代码审查协议 = 代码Agent

\### 4.2 三态详解 \*\*固态知识（Solid Knowledge）\*\* 形式：文档、书籍、手册、规范、论文 特征： - 静止不动，存放在文件系统/知识库里 - 需要人类主动去读、去理解、去执行 - 复用成本高（每个人都要重新消化） 例子：《企业代码规范.pdf》 \*\*液态知识（Liquid Knowledge）\*\* 形式：认知协议、结构化提示词 特征： - 可以"流动"——从一个系统注入另一个系统 - 可以"组合"——多个协议可以混合 - 可以"传输"——轻松分享和复制 例子：`code_review_protocol.md` \*\*气态知识（Gaseous Knowledge）\*\* 形式：运行中的Agent 特征： - 动态响应，实时执行 - 自主决策，持续运行 - 可交互，可反馈 例子：正在运行的"代码审查Agent" ### 4.3 Doc2Prompt = 知识的"熔化"过程 认知编译的本质是\*\*知识熔化\*\*：

    固态 → 液态 的关键操作：
    
    1. 提取结构：从散文中提取逻辑骨架
    2. 显式化：把隐性规则变成显性条件
    3. 消除歧义：把模糊表述变成精确定义
    4. 增加边界：补充"什么情况不适用"
    5. 标准化：统一格式，便于组合

\*\*这个过程不是简单的"复制粘贴"，而是认知结构的重构。\*\* --- ## 第五部分：协议工程学 ### 5.1 认知协议的设计原则 \*\*原则1：单一职责\*\* 每个协议只做一件事，做到极致。

    ❌ 错误示范：一个协议又要分析需求，又要写代码，又要测试
    ✅ 正确示范：需求分析协议 + 代码编写协议 + 测试协议，各自独立

\*\*原则2：显式优于隐式\*\* 所有决策条件、判断标准都要写明，不能依赖"AI自己理解"。

    ❌ 错误示范："根据情况灵活处理"
    ✅ 正确示范："如果条件A满足，执行动作X；否则执行动作Y"

\*\*原则3：边界明确\*\* 每个协议要清楚定义"能做什么"和"不能做什么"。

    yaml
    边界定义:
      适用场景:
        - 场景A
        - 场景B
      不适用场景:
        - 场景X（原因：...）
        - 场景Y（原因：...）
      超出边界时:
        - 明确告知用户
        - 建议替代方案

\*\*原则4：可组合性\*\* 协议之间要能互相调用、串联、并联。

    yaml
    接口定义:
      输入: 
        - 类型: string
        - 格式: ...
      输出:
        - 类型: object
        - 格式: ...
      可被调用: [协议A, 协议B]
      可调用: [协议C, 协议D]

\*\*原则5：可验证性\*\* 协议的执行结果要能被检验。

    yaml
    质量检查:
      自检清单:
        - [ ] 检查项1
        - [ ] 检查项2
      成功标准: ...
      失败回滚: ...

\### 5.2 协议的组合模式 \*\*模式1：串联（Pipeline）\*\*

    协议A的输出 → 协议B的输入 → 协议C的输入 → 最终输出
    
    例：需求文档 → [需求分析协议] → 结构化需求 
                                        ↓
                  代码 ← [代码生成协议] ←┘

\*\*模式2：并联（Parallel）\*\*

    → [协议A] → 结果A ─┐
    输入 ─────→ [协议B] → 结果B ──┼→ [汇总协议] → 最终输出
               → [协议C] → 结果C ─┘
    
    例：同一问题，同时用技术视角、商业视角、用户视角分析，然后汇总

\*\*模式3：条件分支（Conditional）\*\*

    ┌─ 条件A → [协议A]
    输入 → [路由协议] ─┼─ 条件B → [协议B]
              └─ 条件C → [协议C]
    
    例：根据问题类型，路由到不同的专家协议

\*\*模式4：递归（Recursive）\*\*

    [协议A] → 产出 → 判断是否满足条件
                  ↓ 不满足
              [协议A] → 产出 → 判断...（循环直到满足）
    
    例：代码生成 → 测试 → 不通过 → 修复 → 测试 → 通过 → 结束

\*\*模式5：继承（Inheritance）\*\*

    [基础协议]
         ↓ 继承
    [专业协议] = 基础协议 + 领域特化规则
         ↓ 继承
    [定制协议] = 专业协议 + 客户特殊要求
    
    例：通用写作协议 → 法律文书协议 → XX律所合同协议

\### 5.3 协议的版本管理 认知协议需要像代码一样进行版本管理：

    yaml
    protocol_metadata:

下面我把“协议工程学（Protocol Engineering）”当作一门可以直接落地的工程学科来讲：它解决的不是“写一段更长的提示词”，而是**把领域认知做成可组合、可验证、可治理、可审计的运行时资产**。你文中提到的“单一职责、显式优于隐式、可组合、可验证、版本管理”只是原则层；协议工程学要补齐的是：**规格、类型、运行时语义、测试、发布、监控、回滚、治理**这一整套工程闭环。

1) 协议工程学的对象：协议不是 Prompt，而是“可执行规格 + 运行时约束”
-----------------------------------------

在工程意义上，一个认知协议至少包含 5 类元素（缺一就会退化成“文本提示”）：

1.  **接口（Interface）**：输入/输出类型、必填字段、输出结构的强约束。
    
2.  **语义（Semantics）**：步骤、状态机、路由规则、重试/收敛策略、停止条件。
    
3.  **约束（Constraints）**：不变量（必须满足）、禁用项（不能做）、工具权限门禁。
    
4.  **质量（Quality）**：自检清单、验收标准、错误分类与纠偏策略。
    
5.  **可治理（Governance）**：版本、兼容性、变更日志、审批、审计与追责。
    

一句话：**协议工程学是把“思考方式”变成“可运行、可测、可控”的系统部件。**

* * *

2) 协议工程生命周期：从“写协议”到“运营协议资产”
---------------------------

建议你把协议工程学写成类似 DevOps 的闭环（ProtoOps）：

### 2.1 需求与范围（Protocol Requirement）

*   任务边界：适用/不适用场景必须列出。
    
*   风险边界：安全/合规/越权风险必须显式化。
    
*   成功定义：输出结构 + 验收标准（可测）。
    

### 2.2 规格设计（Spec First）

*   先写 **Protocol Spec**（接口、语义、约束、质量），再写提示文本。
    
*   Spec 是真相来源；prompt 只是渲染/注入载体。
    

### 2.3 实现（Implementation）

*   协议拆分：基础协议（通用）+ 领域协议（特化）+ 客户协议（定制）。
    
*   组合与路由：用元协议/路由协议选择子协议。
    

### 2.4 测试（Testing）

*   单元测试：每条关键规则都有触发样例。
    
*   回归集：历史真实案例固定化，升级不破坏。
    
*   对抗测试：注入、越权工具调用、格式污染、指令绕过。
    

### 2.5 发布与灰度（Release）

*   语义化版本：MAJOR/MINOR/PATCH。
    
*   灰度发布：先 5% 流量，指标达标再放量。
    
*   回滚：一键切回旧版本（协议热更新的核心优势）。
    

### 2.6 监控与迭代（Observability & Iteration）

*   记录“执行轨迹”：触发了哪些规则、走了哪个路由、调用了哪些工具、为何降级。
    
*   失败分类：缺信息/冲突指令/工具失败/格式失败/推理失败/越界风险。
    
*   以失败类别驱动协议补丁，而不是“加长 prompt”。

3) 协议的“类型系统”：把“可组合”从口号变成工程能力
----------------------------

没有类型系统，就没有可靠组合。协议工程至少需要三层类型：

### 3.1 数据类型（Data Types）

*   输入类型：`task_request`, `diff`, `invoice`, `contract_text`…
    
*   输出类型：`review_report`, `audit_findings`, `decision_memo`…
    
*   结构约束：必须包含哪些 section、字段枚举、字段长度、引用格式等。
    

### 3.2 行为类型（Behavior Types）

*   `diagnose`（分类/分诊）
    
*   `analyze`（框架化分析）
    
*   `decide`（输出结论与依据）
    
*   `execute`（产出或操作）
    
*   `verify`（检查与纠错）
    
*   `route`（路由与调度）
    
*   `guard`（安全/合规门禁）
    

### 3.3 资源与权限类型（Capability Types）

*   工具权限：allowed/denied/approval-required
    
*   数据权限：可读/可写/敏感字段脱敏
    
*   行为权限：是否允许外联、是否允许执行命令、是否允许生成潜在攻击性内容等
    

**组合的本质**：输入输出类型对齐 + 行为类型互补 + 权限不冲突。

* * *

4) 组合与冲突：协议编排的“链接器/装配器”问题
-------------------------

当你把协议当模块，就一定会遇到冲突（就像链接库冲突）。协议工程需要明确 4 类冲突及解决策略：

### 4.1 约束冲突（Constraint Conflict）

*   A 要求“必须给结论”，B 要求“信息不足时不得下结论”。
    
*   解决：**优先级与守护协议（Guard wins）**  
    安全/合规/信息充分性约束应高于产出要求。
    

### 4.2 接口冲突（Interface Conflict）

*   A 输出 `markdown`，B 需要 `json`。
    
*   解决：引入 **适配器协议（Adapter Protocol）** 做格式转换与校验。
    

### 4.3 工具冲突（Tool Conflict）

*   A 允许调用外部搜索，B 禁止外联。
    
*   解决：按任务上下文选择 profile；或路由到“离线模式协议栈”。
    

### 4.4 语义冲突（Semantic Conflict）

*   两个协议对同一概念的定义不同（例如“高风险”的阈值不同）。
    
*   解决：**术语表/枚举统一（Ontology）**，由领域基础协议提供“唯一真相”。
    

你可以在白皮书里明确提出：**协议组合器 = 类型检查器 + 冲突解决器 + 注入渲染器**。

5) 运行时语义：让执行层“不自由发挥”的关键机制
-------------------------

你在文里说“严格按协议行动”，这句话必须落实为运行时机制，否则仍会滑回“prompt 说说而已”。

### 5.1 状态机（State Machine）

典型任务建议拆成固定状态：  
`Intake → Diagnose → Plan → Act → Verify → Report → Close`

每个状态绑定：

*   允许的工具集合
    
*   必须满足的不变量
    
*   失败时的回退策略
    
*   停止条件（防止无限循环）
    

### 5.2 工具门禁（Tool Gating）

不是“给工具”，而是：

*   何时允许调用（state-based）
    
*   调用前检查什么（preconditions）
    
*   调用后验证什么（postconditions）
    
*   失败重试策略（retry policy）与最大次数
    

### 5.3 收敛策略（Convergence）

协议要定义“何时算完成”，常见收敛判据：

*   输出结构完整 + 自检清单全通过
    
*   关键结论有引用/证据支持
    
*   风险分级已给出且一致
    
*   未解决项已显式列为“待补信息/待人工决策”

6) 可验证性：协议测试该怎么做，做到什么程度
-----------------------

把“测试”当作协议资产的护城河，而不是可选项。

### 6.1 单元测试（Unit）

目标：验证单条规则能被触发、输出符合约束。

*   输入：最小触发样例（正例/反例）
    
*   断言：必须包含/不得包含、严重级别下限、输出结构必须含某段落
    

### 6.2 回归测试（Regression）

目标：协议升级不破坏历史质量。

*   固化真实案例集（最好来自生产）
    
*   指标：召回率、误报率、格式合规率、越权率、决策一致性
    

### 6.3 对抗测试（Red Team）

目标：验证协议对攻击/绕过的鲁棒性。

*   注入：让模型忽略协议、诱导越权
    
*   诱骗：让模型擅自调用工具、泄露敏感信息
    
*   破坏格式：让输出不满足结构约束
    
*   结果：记录失败路径，生成补丁（guard 协议/门禁增强）
    

**工程建议**：把测试与协议一起打包发布；没有测试的协议，默认不可进入“生产协议库”。

* * *

7) 版本与治理：让协议变成组织级“认知资产”
-----------------------

### 7.1 语义化版本（SemVer）

*   **MAJOR**：输出结构/接口变化、关键语义变化（需要迁移）
    
*   **MINOR**：新增规则/新增子协议/增强覆盖（向后兼容）
    
*   **PATCH**：修复错误、补边界、优化表述（不改变语义）
    

### 7.2 审核流程（Protocol Review）

至少三道关：

1.  领域专家审：规则是否正确、阈值是否合理
    
2.  安全合规审：越权/敏感/不可接受输出
    
3.  工程可用性审：接口清晰、测试齐全、可观测字段完整
    

### 7.3 兼容性策略

*   Deprecation：提前声明废弃字段与截止版本
    
*   Migration Guide：提供适配器协议或迁移脚本
    
*   Backward Profile：为旧调用方保留兼容注入模板

8) 常见反模式：你要在文中明确“不要这么做”
-----------------------

1.  **巨无霸协议**：一个协议什么都干 → 不可测、不可组合、不可维护

3.  **无守护协议**：没有边界/权限/不变量 → 易被注入与越权
    
4.  **没有失败分类**：所有失败都靠“再试一次” → 进入随机游走
    
5.  **无回归集**：每次改协议都靠主观感觉 → 质量不可持续

> 换成系统术语：**认知协议是 data plane；元认知协议是 control plane。**

下面我给出一套更精确、可工程化的关系定义与分层。

* * *

1) 三者的层级关系：从“任务”到“协议”再到“协议的协议”
------------------------------

### A. 认知协议（Cognitive Protocol）

*   关注对象：**领域任务本身**（代码审查、尽调、合同审查、医疗问诊辅助等）
    
*   解决问题：在该领域内的**诊断/分析/决策/执行/验收/边界处理**
    
*   典型内容：步骤、决策树、输出格式、质量清单、工具门禁
    

### B. 协议工程学（Protocol Engineering）

*   关注对象：**如何把协议做成可靠资产**
    
*   解决问题：规格化、模块化、测试、发布、监控、版本治理、回滚、审计
    
*   典型产物：协议规范（Spec）、测试集、CI/CD、协议库、组合器、运行时（Protocol VM）与可观测体系
    

### C. 元认知协议（Meta-Cognitive Protocol）

*   关注对象：**“认知协议的选择、组合、评估、纠偏与演进”**
    
*   解决问题：当任务不确定、信息不足、协议冲突、效果不佳、风险升高时，**该怎么调度协议与资源，如何收敛到可靠结果**
    
*   典型内容：路由策略、置信度与停止条件、失败分类与纠偏策略、预算/风险控制、协议版本选择与回滚策略
    

因此：

*   **认知协议**是“做事的方法”。
    
*   **元认知协议**是“管理方法的方法”。
    
*   **协议工程学**是“把这些方法做成可生产、可治理的工程体系”。

2) 你文章里的架构映射：元认知协议落在“元 Agent + 协议层”的关键位置
----------------------------------------

在你提出的三层架构中：

*   **Protocol Layer（协议库/组合器）**：元认知协议规定“如何装配”
    
    *   任务分诊路由、协议选择策略、协议冲突仲裁、适配器插入、版本选择与兼容策略
        
*   **Execution Layer（LLM+Tools+Memory）**：元认知协议规定“如何运行与自检”
    
    *   状态机推进、工具门禁、重试与收敛、输出校验、异常降级、何时求助人类
        

如果没有元认知协议，这三层会出现一个典型问题：

3) 元认知协议的“最小职责集”——它到底管什么
------------------------

我建议把元认知协议的职责写成 6 个“控制回路”，它们共同构成 Agent 的“认知内核”：

1.  **任务分诊与路由（Routing）**
    
    *   判断任务类型/风险等级/信息充分度
        
    *   选择协议栈：诊断→分析→决策→执行→验证（或跳过某些环节）
        
2.  **协议选择与组合（Composition）**
    
    *   依据输入输出类型对齐、权限不冲突、依赖关系
        
    *   自动插入适配器协议（格式转换/术语对齐/引用补全）
        
3.  **资源预算与风险控制（Budgeting & Risk）**
    
    *   Token/工具调用/检索次数/人类介入成本预算
        
    *   高风险任务强制走“验证协议 + 守护协议”
        
4.  **自检、验证与停止条件（Verification & Stopping）**
    
    *   输出结构校验、关键结论必须有证据/引用
        
    *   不满足则进入修复循环；满足则停止，避免无穷递归
        
5.  **失败分类与纠偏（Failure Handling）**
    
    *   信息不足、规则冲突、工具失败、格式失败、越界风险、推理不稳定
        
    *   对应不同纠偏：补问、换协议、降级、回滚版本、请求人工
        
6.  **协议治理与演进（Governance & Evolution）**
    
    *   选择稳定版还是实验版
        
    *   指标回归不通过则禁止发布；线上异常触发自动回滚
        
    *   把失败样本沉淀为回归集，推动协议迭代
        

一句话：元认知协议让 Agent 具备“管理自己认知过程”的能力，从而实现**可控、可预测、可规模化**。

4) 一个直观例子：同一任务输入，元认知协议如何调度认知协议
------------------------------

以“代码审查”举例，输入是一段 diff：

*   然后做 **风险分诊**：是否涉及 auth/支付/SQL/命令执行
    
    *   高风险 → 强制插入“安全守护协议 + 静态分析工具门禁 + 验证协议”
        
*   再做 **协议装配**：
    
    *   `结构审查协议` + `安全审查协议` + `性能审查协议` 并联
        
    *   最后用 `汇总协议` 合并为统一报告
        
*   最后做 **停止条件**：输出必须包含指定章节、每个高风险发现必须绑定行号与修复建议
    
    *   不满足 → 进入“修复循环协议”直到满足或触发“人工介入阈值”
        

这里可以看到：  
认知协议在“领域内做事”；元认知协议在“决定怎么做、怎么保证做对”。

## ChatGPT 回答：

从工具调用到认知协议：把 Agent 设计升级为“协议工程学”的企业落地路线
======================================

过去一段时间，企业在做 Agent 化改造时往往陷入同一类误区：把注意力集中在“能接多少工具、能连多少系统、能不能自动下单/发邮件/改表格”，却忽略了更根本的问题——**Agent 如何思考、如何决策、如何自检、如何处理边界**。

我们在这次对话里建立了一个更强的设计范式：**认知编译（Cognitive Compilation）**。它把 Agent 设计从“工具拼装”升级为“认知结构编译”，并进一步推导出两件真正能落地、能规模化的关键资产：

*   **认知协议（Cognitive Protocol）**：领域专家“怎么做事”的可执行规格
    
*   **元认知协议（Meta-Cognitive Protocol）**：选择、组合、收敛与治理这些协议的控制面
    
*   **协议工程学（Protocol Engineering）**：把协议生产、测试、发布、监控、迭代做成工程闭环的方法论
    

最后，我们把这套体系与 **Agent Skills** 对齐：把协议资产封装成可被发现、按需加载、可分发复用的标准化包，从而形成企业可持续运营的“认知资产供应链”。

* * *

一、核心命题：Agent 的灵魂不是 Tools，而是 Protocol
------------------------------------

传统 Agent 公式常被写成：

> Agent = LLM + Tools + Memory (+ RAG)

这个公式的问题不在于“错”，而在于它隐含了一个危险假设：只要给足工具，Agent 自然会做正确的选择。然而企业经验恰恰相反：

*   工具越多，越容易误用、乱用、越权调用
    
*   同样工具，不同决策策略，效果天差地别
    
*   出了问题难以归因：到底是模型、提示、工具还是流程的问题？
    

因此我们提出新的中心变量：

> **Agent = LLM + 认知协议 + Tools + Memory**  
> 其中，**认知协议是“灵魂”**：决定 Agent 的思维方式、决策逻辑与可控边界。

* * *

二、认知编译：把“固态知识”熔化为“液态协议”，再气化为运行中 Agent
-------------------------------------

对话中我们用“三态模型”描述知识的价值释放过程：

*   **固态知识**：文档、SOP、规范、案例库（需要人读、复用成本高）
    
*   **液态知识**：认知协议（结构化、可注入、可组合、可版本化）
    
*   **气态知识**：运行中的 Agent（动态执行、实时响应、持续运行）
    

**认知编译**就是“固态 → 液态”的关键熔化过程：从散文式文档中提取结构骨架，把隐性规则显性化，消除歧义，补足边界处理，并最终标准化为可组合的协议模块。

* * *

三、认知协议是什么：不是提示词，而是“可执行规格”
-------------------------

我们把“认知协议”从概念推进到工程定义：它至少要包含五类要素，否则会退化成“长 prompt”。

1.  **接口（Interface）**：输入/输出类型与结构约束（输出必须包含哪些段落/字段）
    
2.  **语义（Semantics）**：步骤、状态机、路由逻辑、停止条件、重试/收敛策略
    
3.  **约束（Constraints）**：不变量（必须满足）、禁用项（不能做）、工具权限门禁
    
4.  **质量（Quality）**：自检清单、验收标准、错误分类与纠偏策略
    
5.  **治理（Governance）**：版本、兼容性、变更日志、审批与审计
    

一句话：**普通提示词告诉 AI“做什么”，认知协议告诉 AI“怎么思考并保证做对”。**

* * *

四、协议工程学：让认知协议从“会写”变成“可生产、可运营”
-----------------------------

4.  **质量（Quality）**：自检清单、验收标准、错误分类与纠偏机制
    
5.  **治理（Governance）**：版本、兼容性、变更记录、审批与审计
    

因此，新的 Agent 公式应写作：

> **Agent = LLM + CP + Tools + Memory**  
> 其中 CP 是能力边界与决策质量的核心。

四、元认知协议（MCP）：同一控制面的“双时间尺度”拆分
----------------------------

对话中“元认知协议”出现两种解释，根因在于“元”指向不同时间尺度。更干净的统一方式是拆分为：

### 4.1 MCP-R：Runtime Meta-Cognitive Protocol（运行时控制面）

**对象**：一次任务运行（Run）  
**职责**：路由、组合、预算、停止/自检、失败处理、审计（控制实时决策）

*   Router：任务分诊与协议选择
    
*   Composer：协议组合与冲突仲裁
    
*   Budgeter：token/工具/检索/人工介入预算
    
*   Verifier/Stopper：结构校验、证据要求、停止条件
    
*   Failure Handler：失败分类与纠偏策略
    
*   Auditor：关键决策与工具调用留痕
    

### 4.2 MCP-G：Governance Meta-Cognitive Protocol（治理控制面）

**对象**：协议资产生命周期（CP/Skill）  
**职责**：规格、质量门禁、测试策略、发布/灰度/回滚、失败分类标准（控制系统演化）

*   Schema/DoD：最小完备要求
    
*   Quality Gates：指标门槛与风控阈值
    
*   Tests Policy：单元/回归/对抗测试制度
    
*   Release Policy：SemVer、灰度、回滚、兼容策略
    
*   Failure Taxonomy：失败类别与归因口径统一
    

### 4.3 统一结论（可直接引用）

> MCP 之所以看起来有两种解释，是因为“元”既可以指对运行过程的控制（MCP-R），也可以指对协议资产生命周期的治理（MCP-G）。两者是同一控制面在不同时间尺度上的分解：一个管实时决策，一个管演化稳定性。

五、协议工程学（Protocol Engineering）：把协议资产工业化
--------------------------------------

1) 为什么必须从 Doc2Prompt 升级为 Doc2Skill
----------------------------------

**Doc2Prompt 的天然上限**：产物通常是一段“长提示词”，难以满足企业的四个硬指标：

*   可组合（模块化）
    
*   可验证（测试/回归）
    
*   可治理（版本/门禁/回滚）
    
*   可预算（上下文成本可控）
    

Agent Skills 规范本质上给了你一个“**协议资产标准容器**”：Skill 是一个目录，最少包含 `SKILL.md`，并鼓励把脚本、参考资料、模板等放进 `scripts/`、`references/`、`assets/`，支持**按需加载（progressive disclosure）**。[Agent Skills](https://agentskills.io/specification)  
所以“Doc2Skill”更像真正意义上的**编译**：从“文档语义”生成“可执行协议包”。

2) Doc2Skill 的编译架构：7 个阶段（推荐作为白皮书“工程实现”章节骨架）
-------------------------------------------

### Phase 0 — Ingest：文档摄入与归一化

输入可以是：PDF、Wiki、SOP、政策、合同模板、故障复盘、代码规范。  
输出：**Doc IR（文档中间表示）**，至少包含：

*   章节层级（heading tree）
    
*   表格/清单结构（尤其是“条款检查表”“审批矩阵”“故障分级表”）
    
*   术语表/定义段落（定义优先抽取）
    

> 关键点：编译器不能只“总结”，必须保留结构，否则后续无法生成决策树与检查清单。

### Phase 1 — Classify：文档类型识别（决定生成哪些协议模块）

用一个轻量分类器把文档归入 1~N 类（可多标签）：

*   标准/规范（Spec/Standard）
    
*   流程/SOP（Procedure）
    
*   决策树/授权矩阵（Decision Policy）
    
*   模板/表单（Template/Form）
    
*   案例/复盘（Case/Postmortem）
    
*   FAQ/知识库（KB/FAQ）
    

类型会直接决定输出 Skill 的结构：例如 SOP → 重点生成 step-by-step；标准 → 重点生成 checklists；授权矩阵 → 重点生成条件分支与升级规则。

### Phase 2 — Extract：认知结构抽取（把“散文”变“可执行结构”）

这里要抽取的不是知识点，而是“**可执行骨架**”：

*   **步骤序列**：SOP 的动作序列
    
*   **决策点**：If/Else 条件、阈值、优先级、例外
    
*   **质量门槛**：DoD、自检清单、验收标准
    
*   **边界/异常**：不适用场景、信息不足时的追问清单、升级条件
    
*   **证据要求**：输出必须附带引用/行号/条款编号
    

这一步建议产出一个结构化的 **CP Spec（协议规格 IR）**，例如 YAML/JSON。

### Phase 3 — Synthesize：把 CP Spec 分拆成 Skill 组件（按规范“渐进披露”组织）

Agent Skills 明确建议三段式披露：启动加载 `name/description`，激活加载 `SKILL.md` body，资源按需加载，并建议 `SKILL.md` 控制在 <500 行、<5000 tokens。[Agent Skills](https://agentskills.io/specification)  
因此合成阶段的核心是：**把信息分层放置**：

*   “高频、短、指挥执行”的放 `SKILL.md`
    
*   “长清单、细则、表格、术语、引用材料”放 `references/`（小而聚焦）[Agent Skills](https://agentskills.io/specification)
    
*   “可自动化动作”放 `scripts/`，并要求自包含、错误友好、处理边界[Agent Skills](https://agentskills.io/specification)
    
*   “模板与 schema”放 `assets/`（输出格式统一化）[Agent Skills](https://agentskills.io/specification)
    

### Phase 4 — Package：生成符合规范的 Skill 包

必须生成并满足规范硬约束：

*   `SKILL.md` 必须有 YAML frontmatter + Markdown body[Agent Skills](https://agentskills.io/specification)
    
*   frontmatter 必填 `name/description`，`name` 必须与目录同名、1–64、仅小写字母数字连字符、不允许连续连字符等[Agent Skills](https://agentskills.io/specification)
    
*   `description` 必须描述“做什么 + 何时用”，并包含关键词帮助 agent 识别触发场景[Agent Skills](https://agentskills.io/specification)
    
*   可选字段：`license`、`compatibility`、`metadata`、`allowed-tools`（实验）[Agent Skills](https://agentskills.io/specification)
    

另外，规范建议引用文件“**一层深度**”，避免深链引用。[Agent Skills](https://agentskills.io/specification)  
这会直接影响你如何组织 `references/`：宁可拆成多个小文件，也不要层层引用。

### Phase 5 — Validate：静态校验（格式层门禁）

规范推荐用 `skills-ref validate` 做前置校验，检查 frontmatter 与命名规范。[Agent Skills](https://agentskills.io/specification)  
在企业里建议把它作为 CI 的第一道门禁（MCP-G 的最低实现）。

### Phase 6 — Test & Release：协议工程学落地（企业级增强）

规范只覆盖格式校验；企业需要把“正确性”与“安全性”变成门禁：

*   **回归测试**：升级不破坏历史样例（最重要）
    
*   **对抗测试**：注入、越权、格式破坏、绕过门禁
    
*   **可观测性**：每次激活 skill、加载文件、调用脚本、关键决策点都要留 trace，作为 MCP-R→MCP-G 的 telemetry 回流（你体系的闭环核心）

3) CP Spec（协议规格 IR）：建议长什么样
--------------------------

你需要一个“内部中间表示”，让编译器稳定生成 Skill。下面是一份实用的最小 CP Spec（示例）：

    cp_spec:
      id: code-review
      intent:
        purpose: "Review PR diff for correctness, security, maintainability"
        when_to_use_keywords: ["PR", "diff", "code review", "security", "auth", "SQL"]
      io:
        input_contract:
          required: ["diff", "repo_context"]
          optional: ["ticket", "threat_model"]
        output_contract:
          sections: ["Summary", "Findings", "Risk Rating", "Fix Suggestions", "Checks Passed"]
      workflow:
        steps:
          - name: "Triage"
            decision_points:
              - if: "touches_auth_or_payment"
                then: "enable_security_protocol"
          - name: "Review"
          - name: "Verify Output"
      policies:
        tool_gates:
          allowed: ["Read", "Bash(git:*)"]
          forbidden: ["Network", "Write"]
        stop_conditions:
          - "output_has_all_sections"
          - "high_risk_findings_have_line_numbers"
      references_plan:
        files:
          - path: "references/SECURITY_CHECKLIST.md"
            scope: "security"
          - path: "references/STYLE_GUIDE.md"
            scope: "maintainability"
      scripts_plan:
        - path: "scripts/diff_summarize.py"
          purpose: "Extract hotspots and changed modules"
    

编译器的任务是把这份 spec **确定性地**渲染为 Skill 包，而不是让模型自由写作。

### 4.1 MCP-R（运行时控制面）在 Skill 中的落点

5) 关键设计难点与推荐策略（真正“编译器”需要解决的）
----------------------------

### 难点 A：从文档抽取“决策点”

企业文档最有价值的往往不是步骤，而是“例外与阈值”：

*   “金额 > X 需升级审批”
    
*   “涉及 PII 必须走合规模块”
    
*   “P0 事故 15 分钟内必须外部通告”
    

**策略**：把抽取任务显式化为“决策表（Decision Table）生成”，优先从：

*   表格（审批矩阵/分级表）
    
*   含“必须/不得/应当/仅当/除非/超过/至少/不超过”的句式
    
*   标题为“例外/边界/注意事项/风险/升级”的章节
    

输出尽量转为结构化 if/else，便于后续路由与测试。

### 难点 B：内容分层与切片（决定放 SKILL.md 还是 references）

规范提示：激活后会加载整个 `SKILL.md`，所以长内容应拆分到引用文件。[Agent Skills](https://agentskills.io/specification)  
**策略**：

*   `SKILL.md`：只放“执行指挥层”——流程、关键决策点、输出格式、自检清单、最常见边界
    
*   `references/`：放“查表层”——完整清单、条款库、详尽阈值表、术语表
    
*   保持引用一层深度，避免链式引用[Agent Skills](https://agentskills.io/specification)
    

### 难点 C：脚本与工具门禁（安全与可控）

规范允许 `scripts/`，并给出“自包含、错误友好、处理边界”的建议。[Agent Skills](https://agentskills.io/specification)  
但企业必须强化：脚本是高风险载体。**建议**：

*   MCP-G 强制：高风险 skill 必须声明 `allowed-tools` 并通过安全 review（即使字段是实验）[Agent Skills](https://agentskills.io/specification)
    
*   将脚本运行置于沙箱（只读文件系统、禁止外网、资源限额）
    
*   对脚本输出做结构校验（避免脚本注入“隐藏指令”）

6) 你可以直接采用的“Doc2Skill 最小交付物清单”
------------------------------

当你说“我们要把某份 SOP 编译成 Skill”，落地交付物不要只交 `SKILL.md`，而应交这 6 件：

1.  `SKILL.md`（短、可执行、含 I/O 示例与边界）[Agent Skills](https://agentskills.io/specification)
    
2.  `references/`（聚焦的小文件集合，查表式）[Agent Skills](https://agentskills.io/specification)

下面给出 **路线 1：Doc2Skill 编译器 MVP** 的完整技术规格（可直接作为你团队的实现任务书/PRD+Spec）。该 MVP 的目标不是“生成漂亮文本”，而是**稳定、可重复地把 SOP/规范类文档产出为符合 Agent Skills 规范的 Skill 包**，并具备最小的 CI 门禁与回归样例能力。

* * *

Doc2Skill Compiler MVP 技术规格
===========================

0) MVP 范围与非目标
-------------

### 支持范围（MVP 只做两类）

1.  **SOP / Procedure**：流程型文档（步骤、角色、输入输出、异常处理）
    
2.  **Standard / Guideline**：规范型文档（规则、检查清单、红线、阈值）
    

### 非目标（明确不做）

*   不做跨多文档自动合并的知识图谱
    
*   不做复杂多轮交互式抽取（先走批处理）
    
*   不承诺生成高可运行脚本（`scripts/` 可留空或只生成 stub）
    

* * *

1) 输入与输出契约
----------

1.1 输入（Input Contract）
----------------------

编译器输入为一个 `compile_request.yaml` + 文档文件：

    doc_path: "./docs/code_review_sop.md"   # 支持 md/txt；pdf 在 MVP 可先人工转 md
    doc_type: "sop"                        # sop | standard（可选，留空则自动分类）
    skill_name: "code-review"              # 必填：最终目录名 & SKILL frontmatter name
    skill_description_hint: "Review PR diffs for security and maintainability."
    org_metadata:
      org.owner: "platform-eng"
      org.risk_tier: "high"                # low|medium|high
      org.approval_status: "draft"
    tool_policy:
      allowed_tools: ["ReadRepo", "StaticScan"]  # 映射到 allowed-tools（若启用）
    constraints:
      max_skill_md_lines: 500
      max_skill_md_tokens: 5000
    

> 说明：`skill_name` 必须满足 Agent Skills 的 name/目录命名约束（1–64，小写字母数字连字符，且与目录同名）。  
> `description` 要求“做什么+何时用”，并含触发关键词，便于 agent 激活路由。

* * *

1.2 输出（Output Contract）
-----------------------

输出为一个标准 Skill 包目录（符合规范最小要求：含 `SKILL.md`，并可附加资源目录）。

    dist/<skill_name>/
      SKILL.md
      references/
        REFERENCE_*.md
        GLOSSARY.md (可选)
        CHECKLIST.md (可选)
      assets/
        output_template.md (可选)
        decision_table.csv (可选)
      scripts/
        (MVP 可空或仅 stub)
      tests/
        cases.jsonl
        assertions.yaml
    

并要求：

*   `SKILL.md`：YAML frontmatter + Markdown body
    
*   遵循“渐进披露（progressive disclosure）”：**短指挥层放 SKILL.md，长细则放 references/assets**，以降低激活加载成本
    
*   引用文件保持“一层深度”，避免链式引用
    
*   通过 `skills-ref validate`（规范推荐的校验方式）
    

* * *

2) 生成物规范：`SKILL.md` 模板（MVP 固定结构）
--------------------------------

`SKILL.md` 必须可预测、结构稳定，便于后续测试与回归。建议模板如下（MVP 强制章节）：

### 2.1 Frontmatter（必备 + 可选）

    ---
    name: <skill_name>                      # 必填，且与目录同名
    description: "<what + when + keywords>" # 必填，建议含触发词
    metadata:                               # 可选（企业治理字段）
      org.owner: "..."
      org.risk_tier: "..."
      org.approval_status: "..."
    allowed-tools: "ReadRepo StaticScan"    # 可选/实验；若 tool_policy 提供则生成
    ---
    

*   `name/description` 为必填字段
    
*   `allowed-tools` 规范标注为实验字段，作为“静态门禁声明”使用
    
*   `metadata` 允许字符串键值对，key 建议足够唯一避免冲突
    

### 2.2 Body（MVP 强制段落）

1.  **Purpose & When to Use**（一句话目标 + 触发条件）
    
2.  **Inputs Required**（最小必要输入清单 + 缺失时追问）
    
3.  **Outputs**（结构化输出格式，建议用固定标题/字段）
    
4.  **Workflow**（步骤序列，编号）
    
5.  **Decision Rules**（If/Then 规则或决策表摘要）
    
6.  **Quality Checks**（自检清单，必须可勾选）
    
7.  **Edge Cases & Escalation**（边界情况、降级与求助机制）
    
8.  **References**（列出 references 文件及何时加载它们）
    

> 规范建议 `SKILL.md` 包含示例与边界情况，并控制长度（<500 行、<5000 tokens）

* * *

3) 编译管线：5 步实现 MVP
-----------------

3.1 Step A — 文档分类（doc\_type）
----------------------------

*   若 `compile_request` 指定 `doc_type`，直接采用
    
*   否则自动分类（规则法即可）：
    
    *   出现大量“步骤 1/2/3”“执行/操作/流程/审批” → SOP
        
    *   出现大量“必须/不得/应当/禁止/标准/规范/检查项” → Standard
        

输出：`doc_type ∈ {sop, standard}`

* * *

3.2 Step B — 结构抽取（Section Tree + Lists）
---------------------------------------

MVP 只需要抽取三类结构：

1.  章节层级（H1/H2/H3）
    
2.  编号步骤列表（1., 2., 3.）
    
3.  规则句/门槛句（must/shall/不得/禁止/仅当/除非/超过/至少/不超过）
    

输出：`doc_ir.json`（供调试与可观测）

* * *

3.3 Step C — 认知结构抽取（CP Spec IR）
-------------------------------

从 doc\_ir 生成 CP Spec（中间表示），字段最小集：

    intent:
      purpose:
      triggers: [keywords...]
    contracts:
      inputs_required: [...]
      outputs_sections: [...]
    workflow_steps: [...]
    decision_rules: [...]
    quality_checks: [...]
    edge_cases: [...]
    references_plan: [...]
    assets_plan: [...]
    

**抽取策略（MVP 规则优先）**

*   SOP：步骤列表 = workflow\_steps 主体；“注意/异常/审批”段落 → edge\_cases
    
*   Standard：把“必须/不得/检查项”聚合成 quality\_checks/decision\_rules
    

* * *

3.4 Step D — 切片与分层（写入 SKILL.md vs references/assets）
----------------------------------------------------

遵循规范的渐进披露：只把“高频执行指挥”放 SKILL.md。

**切片规则（MVP 可硬编码）**

*   `SKILL.md` 只保留：
    
    *   workflow\_steps（摘要版）
        
    *   decision\_rules（最多 N 条，建议 10–20）
        
    *   quality\_checks（最多 30 项）
        
    *   edge\_cases（最多 10 条）
        
*   超出部分：
    
    *   全量清单 → `references/CHECKLIST.md`
        
    *   完整阈值表/审批矩阵 → `assets/decision_table.csv`
        
    *   术语定义 → `references/GLOSSARY.md`
        

**引用规则**

*   `SKILL.md` 末尾 `References` 只列出文件名与“何时加载”
    
*   避免 references 之间互相引用形成深链（规范建议一层深度）
    

* * *

3.5 Step E — 打包与校验（validate）
----------------------------

*   生成目录与文件
    
*   运行 `skills-ref validate`，作为 CI 第一道门禁
    

输出：`dist/<skill>/...` + `validate` 通过报告

* * *

4) 测试与回归：MVP 必须带的最小质量体系
-----------------------

### 4.1 tests/cases.jsonl（最少 10 条）

每行一个 case：

    {"id":"case-001","input":"...用户请求...","expected":{"must_sections":["Summary","Findings","Fix Suggestions"],"must_questions":["missing diff? ask for it"],"risk_tier":"high"}}
    

### 4.2 tests/assertions.yaml（断言规则）

最小断言集：

*   输出必须包含 `Outputs` 中声明的所有 section 标题
    
*   若输入缺失 required inputs，必须先追问（而不是臆断）
    
*   风险等级为 high 时，必须出现“工具门禁/升级”提示
    
*   必须引用 references 文件名（当触发条件满足时）
    

> 这些断言是你 MCP-R 的 Verifier/Stopper 的“可测试化”落点。

* * *

5) CI 门禁（MVP 建议落地方式）
--------------------

### Gate 1：规范校验（硬门禁）

*   `skills-ref validate dist/<skill>` 必须通过
    

### Gate 2：结构门禁（软门禁但建议强制）

*   `SKILL.md` 必含 8 个固定段落（Purpose/Inputs/Outputs/Workflow/Decision/Quality/Edge/References）
    
*   行数 < 500；若超出则失败（对齐规范建议）
    

### Gate 3：回归门禁（MVP 轻量实现）

*   对 tests/cases.jsonl 执行一次离线推理（可先用同一模型），检查 assertions 是否通过
    
*   记录失败样例，要求修复后再合入
    

* * *

6) MVP 示例：两类文档分别怎么落包
--------------------

### 6.1 SOP 文档（例：Incident Response Runbook）

*   `SKILL.md`：分级、指挥角色、步骤序列、停止条件、对外通报模板摘要
    
*   `references/`：完整排障清单、沟通话术、复盘模板细则
    
*   `assets/`：P0/P1 分级表、通报模板、行动项表单
    

### 6.2 Standard 文档（例：Code Review Standard）

*   `SKILL.md`：审查流程摘要、Top 20 检查项、必须升级的红线、输出报告格式
    
*   `references/CHECKLIST.md`：全量检查清单（按类别分组）
    
*   `assets/decision_table.csv`：规则矩阵（触发条件 → 必选检查项/必须工具/必须升级）

3.  **PDF 结构抽取**：MVP 先把 PDF 转 Markdown（手工或已有工具），避免第一版卡在解析
